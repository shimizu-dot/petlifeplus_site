1 システム全体構成

システム構成図
レイヤー構成の説明
レイヤー 役割 主なクラス / ファイル
Spring Security 認証・認可ゲートウェイ。セッション管理・CSRF保護・ロール別URLアクセス制御 SecurityConfig, UserDetailsServiceImpl, LoginUser
Controller HTTP リクエストの受付・バリデーション・Service 呼び出し・View/リダイレクト返却 DashboardController, PetController, AppointmentPageController, DatabaseBackupController ...

外部API OpenAI（AI症状チェック）／ Slack・LINE（チャットbot）／ Zoom（オンライン診療） 環境変数で設定。未設定時はフォールバック動作またはスキップ

3 画面遷移フロー

ロール別 画面遷移図

実装準拠ポイント:
URL認可は SecurityConfig に基づく。/app/appointments・/app/calendar は USER/VET/STAFF のみ、/app/consultations は VET/STAFF のみ、/app/admin/users は ADMIN/VET/STAFF、/app/admin/** と /app/reports/** は ADMIN のみ。

4 データフロー

フォーム入力 → Entity → DB の変換フロー
段階クラス / 型主なフィールド変換手段
① 画面入力 GeneralAppointmentForm
PetForm など		 petId(Long), scheduledAt(LocalDateTime), note(String), channel(VISIT/ONLINE)
name/species/birthDate など		 Thymeleaf バインド
@ModelAttribute + Bean Validation（@Valid）
② Entity PetEntity, AppointmentEntity
（Java record）		 AppointmentEntity: petId, ownerUserId, scheduledAt, channel, status ...
PetEntity: ownerUserId, name, species, birthDate, imagePath ...		 Service 内で手動マッピング
Form は日時型/日付型で受け取り、業務チェック後に Entity 化
③ Mapper PetMapper, AppointmentMapper
（MyBatis @Insert/@Update）		 アノテーションSQLで Entity / DTO を直接バインド
insertReturningId / findById で永続化結果を再取得		 MyBatis アノテーション + JDBC
map-underscore-to-camel-case=true
④ DB テーブル pets, appointments など owner_user_id BIGINT, birth_date DATE, deleted_at TIMESTAMP PostgreSQL
GENERATED BY DEFAULT AS IDENTITY
AI症状チェック データフロー（SymptomCheckService）

実装準拠ポイント:
owner/スタッフ権限チェック、永眠ペット操作不可、canUseAiSymptom または ADMIN でAI有効、APIキー未設定/失敗時は fallback-local、保存は insertReturningId。

5 認証・認可フロー

Spring Security 認証フロー（ログイン処理）

実装準拠ポイント:
USERは status=ACTIVE かつ有効サブスクリプションが必要（enabled判定）。認可は SecurityConfig の requestMatchers を上から順に評価し、最後に /app/** は authenticated、その他は permitAll。
