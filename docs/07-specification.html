1 システム概要

システムの目的 必須
このシステムが解決する課題と提供する価値

サービス名　: ペットライフプラス（Pet Life Plus）

利　用　者　: 管理者、一般ユーザー

目　　的　　: ペットの健康記録、AI症状チェック、予約、診療・相談履歴を一元管理し、次の価値を提供する。

・早期異変の発見

・受診判断の支援

・継続的な健康管理の実現

システム構成図 必須
フロントエンド ↔ バックエンド ↔ データベースの関係を図示

フロントエンド: HTML / CSS / JavaScript（静的ファイル、frontend/public/ に配置）

テンプレートエンジン: Thymeleaf（バックエンド側サーバーサイドレンダリング）

バックエンド: Spring Boot（Controller → Service → Mapper）

データベース: PostgreSQL

フロントエンド

HTML / CSS / JavaScript

Thymeleaf テンプレート

HTTPリクエスト ↔ レスポンス
バックエンド

Spring Boot

Controller → Service → Mapper (MyBatis)

データ永続化
データベース

PostgreSQL

ユーザー・ペット・健康記録を管理

利用者はフロントエンド画面を操作し、バックエンドで業務処理を行い、永続データはデータベースに保存する構成です。

F-001 ログイン認証

機能概要 必須
機能ID：F-001

メールアドレス/パスワードで認証し、成功時にセッションを発行してダッシュボードへ遷移する。

画面レイアウト

メールアドレス入力、パスワード入力、ログインボタン、再設定リンクを配置。

処理フロー

1. 認証情報を入力する。

2. ログインボタン押下で認証処理を実行する。

3. 認証結果に応じて遷移またはエラー表示する。

正常時の動作

認証成功でセッションを作成し、ダッシュボードへ遷移する。

エラー時の動作

バリデーション: 未入力・形式不正を項目下に表示。

システム: 認証処理失敗時に汎用エラーメッセージを表示。

F-003 ユーザー管理（CRUD）

機能概要 必須
機能ID：F-003

ユーザー情報の一覧表示、登録、詳細表示、編集、削除を行う（SUPER/ADMIN は全操作可。VET は閲覧のみ。STAFF は USER のみ登録・編集可、削除不可）。削除はアカウントに紐づくデータが存在しない場合に限って実行でき、その場合は status を無効/停止へ更新する。紐づくデータがある場合は削除不可のエラーを表示する。一覧・編集画面では LINE / Slack / Zoom の連携ステータスアイコンを表示し、Slack User ID / LINE User ID の登録も行う。

画面レイアウト

一覧テーブル（table / list / grid ビュー切替）、検索欄、登録ボタン、詳細/編集/削除操作を配置。各ユーザー行に LINE / Slack / Zoom 連携ステータスチップを表示。プランに機能が含まれ ID が登録済みなら色付き、未登録・プラン外はグレーアウト。

処理フロー

1. 検索または対象ユーザーを選択する。

2. 登録/更新/削除を実行する。

3. 保存後に一覧を再表示する。

正常時の動作

操作結果を保存し、一覧へ反映する。

エラー時の動作

バリデーション: 必須・形式・重複エラーを表示。

システム: 競合/保存失敗時に警告表示する。

画面遷移先

F-006 予約管理（CRUD）

機能概要 必須
機能ID：F-006

診療・相談予約の登録、変更、キャンセルを管理し、空き枠チェックを行う。予約方法は「来院 / オンライン」から選択。SUPER / VET / STAFF は予約を承認（CONFIRMED）または却下（CANCELED）できる。

予約ステータス遷移
ステータス説明操作者
REQUESTEDユーザーが登録した初期状態ユーザー
CONFIRMEDSUPER/VET/STAFF が承認済みSUPER / VET / STAFF
CANCELEDオーナーによるキャンセル、または SUPER/VET/STAFF による却下オーナー / SUPER / VET / STAFF
COMPLETED診療完了SUPER / VET / STAFF
画面レイアウト

予約一覧、日付フィルタ、予約フォーム（来院/オンライン カード型選択）、変更/キャンセル操作を配置。SUPER/VET/STAFF ビューには承認・却下ボタンを表示。STANDARD は来院のみ、PREMIUM のみオンライン選択可（LIGHT は予約機能対象外）。

処理フロー（ユーザー登録）

1. 予約方法（来院 / オンライン）・日時・種別を入力する。

2. 空き枠と重複をチェックする。

3. ステータス REQUESTED で登録し一覧へ反映する。

処理フロー（承認・却下）

1. SUPER/VET/STAFF が予約一覧で対象予約を選択する。

2. 「承認」（POST /app/appointments/{id}/approve）または「却下」（POST /app/appointments/{id}/reject）を実行する。

3. ステータスを更新し最新状態を表示する。

エラー時の動作

バリデーション: 過去日時・必須漏れ・プラン外オンライン選択を表示。

システム: 競合時に警告表示。権限外の承認・却下操作は 400 エラー。

枠の種類is_blocked説明

F-009 AI症状チェック

機能概要 必須
機能ID：F-009

ペットに紐付いた1ターン完結の症状解析機能。症状・発症時期・メモを送信すると重症度（HIGH/MEDIUM/LOW）と推奨対応（VISIT/CONSULT/OBSERVE）を即時返却し、symptom_checks テーブルに保存する。

画面レイアウト

ペット詳細ページ（/app/pets/{id}）内のフォームとして組み込み。結果は同ページへリダイレクトして表示。

処理フロー

1. ペット詳細ページのフォームから症状種別・発症時期・メモを入力。

2. POST /app/pets/{id}/symptom-check で送信。

3. SymptomCheckService が OpenAI API（またはヒューリスティック）で判定。

4. 結果を symptom_checks に保存し、ペット詳細ページへリダイレクト。

正常時の動作

重症度（HIGH/MEDIUM/LOW）と推奨対応（VISIT/CONSULT/OBSERVE）をフラッシュメッセージで表示。直近の診断履歴としてページに反映。

エラー時の動作

バリデーション: 症状種別が空の場合はエラー表示。

システム: APIエラー時はキーワードベースのヒューリスティック判定に自動フォールバック。

画面遷移先

ペット詳細画面（リダイレクト）

予約管理画面（結果から導線）

出力項目: 実施日時・症状の種類・発症時期・重症度（HIGH/MEDIUM/LOW を色分け）・推奨対応（受診/相談/様子見）・AIガイダンス本文。

機能概要 必須
機能ID：F-011　実装済み（BillingController / BillingService / BillingNotificationService）

サブスクリプションの契約更新申請時に請求を自動発行し、顧客へアプリ内通知・メール・LINE で支払い案内を送信する。支払い方法は銀行振り込みまたは PayPay。管理者が入金を登録すると支払状況（UNPAID / PARTIAL / PAID）が更新される。

管理画面: GET /app/admin/billing（一覧）/ GET/POST /app/admin/billing/{id}（詳細・入金登録）— ADMIN / SUPER のみアクセス可。

通知チャンネルと実行方式
チャンネル実行方式条件説明
① アプリ内通知同期常に実行呼び出し元スレッドで実行。通知センター（/app/notifications）に即座に届く。失敗は即検知できる
② メール（SendGrid）非同期（@Async）SENDGRID_API_KEY 設定時別スレッドで実行。遅延・失敗してもアプリ内通知には影響しない
③ LINE Push非同期（@Async）line_user_id 登録済み かつ LINE_CHANNEL_TOKEN 設定時テキスト形式で請求概要を送信

通知再送: 管理者が請求詳細画面の「📨 お支払い案内を再送する」ボタン （POST /app/admin/billing/{id}/resend-notification）で任意のタイミングで再送可能。

画面レイアウト

請求一覧（subscription_id・金額・支払状況・期日）、入金登録フォーム、領収書出力を配置。

処理フロー

1. ユーザーが POST /app/subscriptions/{id}/renew-request で更新申請する。

2. BillingService.createInvoice() が invoices レコードを作成する（支払期限 = 請求日 + 14日）。

3. アプリ内通知を同期送信 → 通知センターに即座に表示。

4. メール・LINE を非同期（@Async）送信 → 別スレッドで実行。

5. 顧客が銀行振り込みまたは PayPay で支払いを行う。

6. 管理者が POST /app/admin/billing/{id}/payments で入金登録する。

7. 累計入金額が請求額以上になれば payment_status を PAID に更新。入金確認通知をアプリ内（同期）+ メール（非同期）で送信する。

エラー時の動作

バリデーション: 金額不正・必須漏れ・支払方法不正を表示。すでに PAID の請求書への入金登録は 400 エラー。

システム: createInvoice() が失敗した場合はエラーメッセージをユーザーに表示（サイレント握りつぶしなし）。各通知チャンネルは独立して catch し、他チャンネルに影響しない。

通知再送: 管理画面の「📨 お支払い案内を再送する」ボタン（POST /app/admin/billing/{id}/resend-notification）で任意のタイミングで再送可能。

F-022 チャットボット相談

機能概要 必須
機能ID：F-022

ユーザーとのマルチターン会話で症状情報を段階的に収集し、受診判断をサポートする機能。F-009（AI症状チェック）とは独立した別機能。ペット紐付けなし。全プランで利用可能。

F-009 との違い
F-009 AI症状チェックF-022 チャットボット相談
会話ターン数1ターン完結マルチターン（履歴50件）
ペット紐付けあり（pet_id）なし（user_id のみ）
利用プランSTANDARD 以上全プラン（無料含む）
AI エンジンOpenAI（openai.*）Groq / fallback（chatbot.*）
画面レイアウト

独立したチャット画面（/app/consult/chatbot）。会話履歴・情報収集フロー進捗（4段階）・クイック入力ボタンを表示。

処理フロー

1. ユーザーがメッセージを入力して送信（POST /app/consult/chatbot）。

2. ConsultChatService がユーザーメッセージを DB に保存。

3. 会話履歴を含めて Groq API（またはキーワードベースフォールバック）に送信し返答を生成。

4. ボット返答を DB に保存してチャット画面へリダイレクト。

エラー時の動作

バリデーション: メッセージが空の場合はエラー表示。

システム: APIエラー / キー未設定時は fallbackReply()（5段階キーワードフロー）で応答。

入力項目一覧 必須
項目名 データ型 必須 バリデーション 備考
メッセージ (message) String ○ @NotBlank / 1000文字以内

5 非機能要件

セキュリティ要件 必須
認証・認可・データ保護に関する要件
  • 認証方式（Spring Security）: フォームログインを基本にSpring Securityで認証・認可を実装し、URL単位でアクセス制御する。
  • パスワードの暗号化方式: BCryptPasswordEncoderでハッシュ化して保存し、平文保存は行わない。
  • CSRF対策: Spring SecurityのCSRF保護を有効化し、フォーム送信時にCSRFトークンを検証する。
  • XSS対策: 入力値バリデーションと出力時エスケープ（Thymeleaf自動エスケープ）を実施する。
  • SQLインジェクション対策: MyBatis Mapperでバインド変数（#{}）を使用し、SQL文字列連結（${}の安易な利用）を禁止する。
  • セッション管理: ログイン時のセッションID再生成、セッションタイムアウト設定、ログアウト時のセッション破棄を実施する。

対応ブラウザ・デバイス
  • 対応ブラウザ: Chrome（最新版）/ Firefox（最新版）/ Edge（最新版）
  • 対応デバイス: PC / タブレット / スマートフォン
