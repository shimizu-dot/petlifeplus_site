# 要件定義まとめ（Pet Life Plus）

## 1. システム概要
- サービス名: ペットライフプラス（Pet Life Plus）
- 利用者: 管理者、一般ユーザー
- 目的: ペットの健康記録、AI症状チェック、予約、診療・相談履歴を一元管理し、早期異変の発見、受診判断の支援、継続的な健康管理を実現する。

## 2. システム構成
- フロントエンド: HTML / CSS / JavaScript（静的ファイル、`frontend/public/` に配置）
- テンプレートエンジン: Thymeleaf（バックエンド側のサーバーサイドレンダリング）
- バックエンド: Spring Boot（Controller → Service → Repository）
- データベース: PostgreSQL

### F-006
機能名：予約管理（CRUD）
機能ID：F-006  診療・相談予約の登録、変更、キャンセルを管理し、空き枠チェックを行う。予約は「来院 / オンライン」の方法を選択できる。VET / STAFF / ADMIN は予約を承認（CONFIRMED）または却下（CANCELED）できる。

> **プランアクセス制限:** 予約ページ（`/app/appointments`）は **STANDARD プラン以上**が必要。Light プランユーザー（一般 USER で AI_SYMPTOM 機能コードなし）は `AppointmentPageController.ensureAccessible()` にて 400 エラーとなる。VET/STAFF/ADMIN/SUPER は制限なし。

#### 予約ステータス遷移

| ステータス | 説明 | 操作者 |
| --- | --- | --- |
| REQUESTED | ユーザーが登録した初期状態 | STANDARD以上のユーザー |
| CONFIRMED | VET/STAFF/ADMIN が承認済み | VET / STAFF / ADMIN / SUPER |
| CANCELED | ユーザーがキャンセル、または VET/STAFF/ADMIN が却下 | ユーザー（cancel） / VET / STAFF / ADMIN / SUPER（reject） |
| COMPLETED | 診療完了 | VET / STAFF / ADMIN / SUPER |

#### エンドポイント一覧
- `GET/POST /app/appointments` — 一覧・新規登録（STANDARD以上 + VET/STAFF/ADMIN）
- `POST /app/appointments/{id}/approve` — 承認 → CONFIRMED（VET/STAFF/ADMIN/SUPER のみ）
- `POST /app/appointments/{id}/reject` — 却下 → CANCELED（VET/STAFF/ADMIN/SUPER のみ）
- `POST /app/appointments/{id}/cancel` — ユーザー自身によるキャンセル（オーナーのみ、スタッフは不可）

#### 予約枠管理（`/app/admin/appointment-slots`）
SUPER/ADMIN/VET/STAFF が自動生成スロット（9:30–17:00 30分刻み）を調整する管理画面。

- **過去日付制限:** 本日より前の日付を指定した場合はエラーメッセージを表示し、スロット一覧・追加枠フォームを非表示にする。日付ピッカーの `min` 属性で本日以降のみ選択可。
- **追加枠（`is_blocked=false`）:** 業務時間外や特別スロットを追加。自動生成スロットに上乗せされる。
- **ブロック枠（`is_blocked=true`）:** 特定の自動生成スロットを非表示化。休診・院内研修等に使用。
- **削除:** 追加枠・ブロック枠は削除ボタンで登録を取り消せる（申請済みの枠は削除不可）。
- `POST /app/appointments/delete-selected` — 選択した予約の一括削除

画面レイアウト
- 予約一覧、日付フィルタ、予約フォーム（来院/オンライン カード型選択）、変更/キャンセル操作を配置。
- VET/STAFF/ADMIN ビューには承認・却下ボタンを表示。オンライン選択時はPREMIUM以上のみ有効（`canUsePrioritySupport()`）。

処理フロー（ユーザー登録）
1. 予約方法（来院 / オンライン）・日時・種別を入力する。
2. 空き枠（9:30-17:00、30分刻みで自動生成）と重複をチェックする。
3. ステータス REQUESTED で登録し一覧へ反映する。

処理フロー（承認・却下）
1. VET/STAFF/ADMIN が予約一覧で対象予約を選択する。
2. 「承認」または「却下」ボタンを押下する。
3. ステータスを更新する。

エラー時の動作（バリデーション/システム）
- バリデーション: 過去日時・必須漏れを表示。オンライン選択時にプラン外ならエラー。
- システム: 競合時に警告表示する。権限外操作は 400 エラー。

画面遷移先
- 予約詳細画面
- 予約編集画面
- 通知配信管理画面

入力項目一覧
| 項目名 | フィールド名 | データ型 | 必須 | バリデーション | 備考 |
| --- | --- | --- | --- | --- | --- |
| 予約日時 | scheduledAt | DateTime | ○ | 必須チェック / 過去日時不可 | |
| 予約種別 | appointmentType | Enum | ○ | 必須チェック | 診療 / 相談 |
| 予約方法 | channel | Enum | ○ | 必須チェック | 来院 / オンライン |

### F-009
機能名：AI症状チェック
機能ID：F-009  ペットに紐付いた1ターン完結の症状解析機能。症状・発症時期・メモを送信すると重症度（HIGH/MEDIUM/LOW）と推奨対応（VISIT/CONSULT/OBSERVE）を即時返却し、`symptom_checks` テーブルに保存する。

画面レイアウト
- ペット詳細ページ（`/app/pets/{id}`）内のフォームとして組み込み。結果は同ページへリダイレクトして表示。

処理フロー
1. ペット詳細ページのフォームから症状種別・発症時期・メモを入力。
2. `POST /app/pets/{id}/symptom-check` で送信。
3. `SymptomCheckService` が OpenAI API（またはヒューリスティック）で判定。
4. 結果を `symptom_checks` に保存し、ペット詳細ページへリダイレクト。

エラー時の動作（バリデーション/システム）
- バリデーション: 症状種別が空の場合はエラー表示。
- APIエラー: 自動的にキーワードベースのヒューリスティック判定にフォールバック。

入力項目一覧
| 項目名 | フィールド名 | データ型 | 必須 | バリデーション |
| --- | --- | --- | --- | --- |
| 症状の種類 | symptomType | String | ○ | @NotBlank @Size(max=100) |
| 発症時期 | onsetText | String | — | @Size(max=100) |
| 補足メモ | memo | String | — | @Size(max=500) |

> 詳細な技術仕様は `backend/docs/ai-features.md` §1 を参照。

**印刷出力への連携**
AI症状チェック履歴は診療情報印刷画面（`GET /app/pets/{petId}/health-records/print`）に自動的に含まれる。出力項目: 実施日時・症状の種類・発症時期・重症度（HIGH/MEDIUM/LOW）・推奨対応（受診/相談/様子見）・AIガイダンス本文。

### F-011
機能名：請求・決済管理
機能ID：F-011  サブスクリプションの新規申し込みおよび契約更新時に請求を発行し、顧客へメール・LINE で支払い案内を送信する。支払い方法は銀行振り込みまたは PayPay。入金確認後に支払状況を更新する。

> **実装状況:** 実装済み（`BillingController` / `BillingService` / `BillingNotificationService`）。コントローラーは `/app/admin/billing/**`（ADMIN/SUPER のみ）。

#### トリガーイベントと通知フロー

| イベント | 発生タイミング | 通知チャンネル | 通知内容 |
| --- | --- | --- | --- |
| 契約更新申請 | ユーザーが `POST /app/subscriptions/{id}/renew-request` を実行 | **アプリ内通知**（必須）+ メール（SendGrid）+ LINE（LINE User ID 登録時） | プラン名、請求金額、支払期日、支払い方法案内 |
| 入金確認完了 | 管理者が `/app/admin/billing/{id}/payments` で入金登録し PAID になった時 | **アプリ内通知**（必須）+ メール（SendGrid） | 受領確認メッセージ |

**通知の実行順序（`BillingNotificationService`）:**
1. **アプリ内通知（同期）** — `sendInvoiceIssuedInApp()` / `sendPaymentConfirmedInApp()` を呼び出し元スレッドで実行。失敗すると例外が上位に伝わり即検知できる。SendGrid・LINE 未設定でも必ず `通知センター（/app/notifications）` に届く
2. **メール + LINE（非同期 `@Async`）** — `notifyInvoiceIssued()` / `notifyPaymentConfirmed()` を別スレッドで実行。遅延・失敗しても①には影響しない

**通知再送:** 管理者が請求詳細画面（`POST /app/admin/billing/{id}/resend-notification`）の「📨 お支払い案内を再送する」ボタンで再送可能。初回通知失敗時や顧客からの確認依頼時に使用する。

#### 支払い方法

| 方法 | DB 値 | 備考 |
| --- | --- | --- |
| 銀行振り込み | `BANK` | 振込先口座は案内メール・LINE に記載 |
| PayPay | `PAYPAY` | PayPay リンクまたは QR コードを案内に添付 |
| その他 | `OTHER` | 将来の拡張用 |

#### DB テーブル構成
- `subscriptions` — プラン契約（`auto_renew` フラグあり）
- `invoices` — 請求書（`payment_status`: UNPAID / PARTIAL / PAID）
- `payments` — 入金記録（`payment_method`: BANK / PAYPAY / OTHER）

画面レイアウト
- 請求一覧（subscription_id・金額・支払状況・期日）、入金登録フォーム、領収書出力を配置。

処理フロー
1. ユーザーが `POST /app/subscriptions/{id}/renew-request` で更新申請する。
2. `BillingService.createInvoice()` が `invoices` レコードを作成する（支払期限 = 請求日 + 14日）。
3. アプリ内通知を**同期**送信（`sendInvoiceIssuedInApp()`）→ 通知センターに即座に表示。
4. メール・LINE を**非同期**送信（`notifyInvoiceIssued()`）→ 別スレッドで実行。
5. 顧客が銀行振り込みまたは PayPay で支払いを行う。
6. 管理者が `POST /app/admin/billing/{id}/payments` で入金登録する。
7. 累計入金額が請求額以上になれば payment_status を PAID に更新。
8. 入金確認通知をアプリ内（同期）+ メール（非同期）で送信する。

エラー時の動作（バリデーション/システム）
- バリデーション: 金額不正・必須漏れ・支払方法不正・PAID 済みへの重複登録を表示。
- システム: `createInvoice()` が失敗した場合はエラーメッセージを表示（サイレント握りつぶしなし）。各通知チャンネルは独立して失敗を catch しており、他チャンネルに影響しない。
- **通知再送:** 管理画面の「📨 お支払い案内を再送する」ボタン（`POST /app/admin/billing/{id}/resend-notification`）で任意のタイミングで再送可能。

画面遷移先
- 請求詳細画面
- 入金詳細画面
- レポート画面

入力項目一覧（入金登録）
| 項目名 | フィールド名 | データ型 | 必須 | バリデーション | 備考 |
| --- | --- | --- | --- | --- | --- |
| 請求金額 | amount | Decimal | ○ | 0 以上 | invoices から引継ぎ |
| 支払金額 | paidAmount | Decimal | ○ | 0 以上 | 分割払い対応 |
| 支払日時 | paidAt | DateTime | ○ | 日時形式 | |
| 支払い方法 | paymentMethod | Enum | ○ | BANK / PAYPAY / OTHER | |
| 振込参照番号 / 取引ID | transactionRef | String | — | 100文字以内 | 銀行振り込みは振込番号、PayPay は取引ID |

## 5. 非機能要件

### セキュリティ要件
- 認証方式（Spring Security）: フォームログインを基本にSpring Securityで認証・認可を実装し、URL単位でアクセス制御する。
- パスワードの暗号化方式: BCryptPasswordEncoderでハッシュ化して保存し、平文保存は行わない。
- CSRF対策: Spring SecurityのCSRF保護を有効化し、フォーム送信時にCSRFトークンを検証する。
- XSS対策: 入力値バリデーションと出力時エスケープ（Thymeleaf自動エスケープ）を実施する。
- SQLインジェクション対策: MyBatis Mapperでバインド変数（`#{}`）を使用し、SQL文字列連結（`${}`の安易な利用）を禁止する。
- セッション管理: ログイン時のセッションID再生成、セッションタイムアウト設定、ログアウト時のセッション破棄を実施する。
