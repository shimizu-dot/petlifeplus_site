# AGENTS.md

## Global Rules (Must Follow)
- 不要な確認質問をしない。指示がある作業はそのまま実行する。
- 作業対象外のセクション・ファイルを触らない。
- 指摘されたレイアウト崩れは、対象要素だけを最小差分で修正する。
- 編集後は必ず `git diff` で意図しない変更がないか確認する。
- 変更・修正を行った場合は、必ず `CHANGELOG.md` に日付と内容を記載する。

## HTML/SVG Editing Rules
- 「実装準拠ポイント:」は文書として記載し、画像外に置く。
- 「実装準拠ポイント:」の直後で改行する（`実装準拠ポイント:<br>...`）。
- 注釈の位置指定がある場合は厳守する（例: 画像のすぐ下、画像内など）。
- 線と文字が重ならないように座標を調整する。
- フォントサイズ変更は指示がない限り他要素へ波及させない。

## PetEntity Section Specific
- 作業中は `design/class-diagram.html` の PetEntity ブロックのみ編集する。
- User.java ブロックには一切追記しない。

## Post-Edit Checklist
1. `実装準拠ポイント:` の直後で改行している。
2. 説明文と罫線が重なっていない。
3. 余白が過剰になっていない。
4. User.java 側へ混入していない。


### サンプル１　チーム開発向け
## 目的
このリポジトリは複数人で保守しています。整合性、安全性、レビューしやすさを重視してください。

## 作業方針
- 問題をきれいに解決できる最小限の変更を優先する。
- 巧妙さより、読みやすさと保守性を優先する。
- 必要がない限り、無関係なリファクタリングは行わない。
- 明確な理由がない限り、既存のプロジェクト慣習を維持する。

## 変更前に行うこと
- 編集前に関連ファイルを読む。
- 既存の実装パターンを確認し、それに合わせる。
- 影響範囲をできるだけ小さく見積もる。
- 要件が曖昧な場合は、作業時の前提を明示する。

## コード変更
- 関数やモジュールは単一責務に保つ。
- 不要な依存関係は追加しない。
- 破壊的変更、大規模な書き換え、スキーマ変更、依存追加は事前に確認を取る。
- 明示的な要求がない限り、公開インターフェースは維持する。
- 振る舞いを変えた場合は、関連するドキュメント、設定、サンプルも更新する。

## テスト
- まずは関連する最小限のテストを実行する。
- 利用可能なら、完了前に lint と format の確認も行う。
- 振る舞いが変わる場合は、テストを追加または更新する。
- テストを実行できなかった場合は、その理由を明示する。

## 安全性とレビュー
- シークレット、トークン、認証情報をコミットしない。
- 危険なマイグレーション、破壊的操作、取り消せない変更は、実行前に明示する。
- 人がレビューしやすい、差分の明確な変更を優先する。

## 出力時の期待
- 何を変更したかを平易に要約する。
- 主要な変更ファイルを挙げる。
- 今後の課題、制約、前提条件を記載する。
- テスト結果、または未検証の点を明記する。


###サンプル２　個人開発向け
## 目的
このリポジトリは個人開発用です。スピード、明快さ、実用的な前進を重視してください。

## 基本方針
- 後から見直しやすい、単純な解決策を優先する。
- 過剰設計を避け、作業速度を保つ。
- リスクが高くない限り、合理的な前提を置いて進める。
- 改善案は歓迎するが、明確な効果がない限りスコープを広げない。

## コードスタイル
- リポジトリ内の既存パターンに従う。
- 分かりやすく率直なコードを優先する。
- 可能な範囲でボイラープレートは減らす。
- 大きな効果がない限り、新しい依存は追加しない。

## 変更方針
- 依頼された成果に集中した変更にする。
- 可読性が上がる程度の小さな整理は行ってよい。
- 大きな設計変更を行う場合は、先にトレードオフを説明する。

## 検証
- テストがあれば、関連するものを実行する。
- テストがなければ、可能な範囲で基本的な動作確認を行う。
- 確認できなかった点は明記する。

## ドキュメント
- 開発体験に関わる変更があれば、README、コメント、サンプルを更新する。
- 後で迷いそうな点には、簡潔な補足を残す。

## 出力時の期待
- 何を変えたかを簡潔に説明する。
- 省略したことや割り切った点があれば述べる。
- 次にやるとよい改善案があれば示すが、必須扱いにはしない。

###サンプル３　Python プロジェクト向け
## 目的
このリポジトリは Python プロジェクトです。正確さ、可読性、保守しやすい構成を重視してください。

## Python の方針
- 既存のディレクトリ構成、命名規則、実装パターンに従う。
- 巧妙な近道より、明快で明示的なコードを優先する。
- 関数は小さく、責務を分ける。
- コードベースで使われている場合、または可読性向上に有効な場合は、型ヒントを付ける。
- 公開関数やクラスで意図が自明でないものには docstring を付ける。

## 依存関係
- 妥当な範囲では標準ライブラリを優先する。
- 不要なパッケージは追加しない。
- 重い依存の導入、パッケージ管理方式の変更、lockfile の更新は事前に確認を取る。

## コード品質
- import は整理して保つ。
- import 時に隠れた副作用を持ち込まない。
- 失敗しうる処理では、例外やエラーを明示的に扱う。
- 観測性に役立つ場合は logging を使う。ただし過剰な出力は避ける。

## テスト
- 振る舞いが変わる場合は、テストを追加または更新する。
- まずは対象範囲の狭いテストを優先し、必要に応じて広い確認を行う。
- pytest を使っている場合は、関連する pytest を実行する。
- linter や formatter がある場合は、可能な範囲で実行する。

## データとスクリプト
- 破壊的なファイル操作やデータ移行は慎重に扱う。
- スクリプトは可能なら冪等性を意識する。
- シークレット、ローカル環境依存のパス、環境固有値を直書きしない。

## ドキュメント
- セットアップ、CLI の挙動、設定、公開 API を変えた場合は README や開発者向け文書も更新する。
- 新しいスクリプトやコマンドには、必要に応じて使用例を加える。

## 出力時の期待
- コード変更の内容を明確に要約する。
- 実行したテスト、lint、各種チェックを報告する。
- 前提条件、未実施の確認、残るリスクを明記する。
